# Research: Node 24 + pnpm Migration

## Decision: pnpm version pinning via `packageManager` field

**Decision**: Use the `packageManager` field in `package.json` (Corepack standard) to pin the
exact pnpm version used at migration time.

**Rationale**: Corepack (bundled with Node 16+, included in Node 24) reads `packageManager` and
automatically downloads the declared version if needed. This means contributors don't need to
separately install pnpm — `corepack enable` is sufficient.

**Alternatives considered**:
- `.npmrc` with `packageManager` — not a standard field, less tooling support
- Documenting version in README only — not enforced

---

## Decision: `engine-strict=true` in `.npmrc`

**Decision**: Add `browser/.npmrc` with `engine-strict=true`.

**Rationale**: Without this, `npm install` on the repo will silently succeed despite the
`engines` field declaring Node 24+. `engine-strict=true` converts the warning into a hard
failure, preventing accidental npm usage.

**Alternatives considered**:
- `preinstall` script that checks for pnpm — more complex, same effect
- README note only — not enforced

---

## Decision: Unpin vitest/happy-dom to latest

**Decision**: Restore `vitest`, `@vitest/coverage-v8`, and `happy-dom` to their pre-pin
`^latest` semver ranges (vitest `^4.x`, happy-dom `^20.x`).

**Rationale**: The Node 18 pins (vitest 1.x, happy-dom 14.x) were a workaround. On Node 24 the
latest versions are fully supported and include bug fixes and performance improvements.

**Alternatives considered**:
- Keep pinned — defeats the purpose of upgrading
- Use exact versions — too rigid, misses patch releases

---

## Decision: Delete `package-lock.json`, commit `pnpm-lock.yaml`

**Decision**: Remove `browser/package-lock.json` from the repo, add it to `.gitignore`, and
commit the `pnpm-lock.yaml` generated by `pnpm install`.

**Rationale**: Having both lockfiles creates ambiguity about which package manager to use.
Committing `pnpm-lock.yaml` ensures reproducible installs. Gitignoring `package-lock.json`
prevents it from reappearing if someone accidentally runs npm.

**Alternatives considered**:
- Keep `package-lock.json` alongside `pnpm-lock.yaml` — causes confusion, npm vs pnpm divergence
- Neither lockfile committed — not reproducible
